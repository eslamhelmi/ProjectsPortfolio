Demand Forecasting & Inventory Optimization

IIF Certified Project | Lancaster University CMAF Program

Overview

This project involved developing a comprehensive demand forecasting and inventory optimization framework for 20 consumer goods products across four distinct demand patterns. Using rolling origin evaluation and a hierarchical method selection framework, I identified optimal forecasting methods for each product and calculated safety stock levels for a 97% service level.

Project Scope

  • 20 daily time series (70 days each)
  • Four product families (A, B, C, D)
  • Multiple demand patterns (smooth to intermittent)
  • 1-week lead time
  • 97% target service level (z = 1.88)

Tools & Methods

  • R (forecast, smooth, tsintermittent)
  • ETS, MA, SES, Naive (standard methods)
  • Croston, SBA, TSB (intermittent methods)
  • Rolling origin evaluation (15 origins)
  • RMSE, MASE, ME metrics

# DEMAND CLASSIFICATION USING CV AND ZERO PERCENTAGE

# Function to calculate classification metrics
classify_demand <- function(y) {
  cv <- sd(y) / mean(y)                    # Coefficient of variation
  pct_zero <- sum(y == 0) / length(y) * 100  # Percentage of zeros
  
  # Classification rules (Syntetos-Boylan framework adapted)
  if (pct_zero > 30) {
    pattern <- "Intermittent"
  } else if (cv > 0.8) {
    pattern <- "Erratic"
  } else if (cv > 0.5) {
    pattern <- "Variable"
  } else {
    pattern <- "Smooth"
  }
  
  return(data.frame(CV = cv, PctZero = pct_zero, Pattern = pattern))
}

# Apply to all products
classification <- do.call(rbind, lapply(names(data)[-1], function(p) {
  result <- classify_demand(data[[p]])
  result$Product <- p
  result$Family <- substr(p, 1, 1)
  result
}))

2. Rolling Origin Evaluation Framework

# ============================================================
# ROLLING ORIGIN EVALUATION FOR STANDARD DEMAND (Families A, B, C)
# ============================================================

evaluate_standard <- function(data, product_name, origins, h, first_origin) {
  # Extract time series with weekly frequency
  y <- ts(data[[product_name]], frequency = 7)
  
  # Methods to evaluate
  methods <- c("Naive", "MA", "ETS_Restricted")
  n_methods <- length(methods)
  n_origins <- length(origins)
  
  # Initialize matrices to store results
  rmse_matrix <- matrix(NA, nrow = n_origins, ncol = n_methods,
                        dimnames = list(paste0("o", origins), methods))
  me_matrix <- matrix(NA, nrow = n_origins, ncol = n_methods,
                      dimnames = list(paste0("o", origins), methods))
  mae_matrix <- matrix(NA, nrow = n_origins, ncol = n_methods,
                       dimnames = list(paste0("o", origins), methods))
  
  # Loop through each origin
  for (i in seq_along(origins)) {
    origin <- origins[i]
    
    # Split data: expanding window
    train <- y[1:origin]
    actual <- as.numeric(y[(origin + 1):(origin + h)])
    
    # ----- NAIVE -----
    fc_naive <- as.numeric(naive(train, h = h)$mean)
    
    # ----- MOVING AVERAGE (k=7) -----
    fc_ma <- as.numeric(forecast(sma(train, silent = TRUE), h = h)$mean)
    
    # ----- ETS WITH RESTRICTED POOL -----
    # Only consider additive models to avoid numerical issues
    fc_ets <- as.numeric(forecast(adam(train,
                  model = c("ANN","ANA","MNN","MNM","AAdN","AAdA","MAdN","MAdM"),
                  silent = TRUE), h = h)$mean)
    
    # Calculate metrics for THIS origin
    errors_naive <- actual - fc_naive
    me_matrix[i, "Naive"] <- mean(errors_naive)
    rmse_matrix[i, "Naive"] <- sqrt(mean(errors_naive^2))
    mae_matrix[i, "Naive"] <- mean(abs(errors_naive))
    
    errors_ma <- actual - fc_ma
    me_matrix[i, "MA"] <- mean(errors_ma)
    rmse_matrix[i, "MA"] <- sqrt(mean(errors_ma^2))
    mae_matrix[i, "MA"] <- mean(abs(errors_ma))
    
    errors_ets <- actual - fc_ets
    me_matrix[i, "ETS_Restricted"] <- mean(errors_ets)
    rmse_matrix[i, "ETS_Restricted"] <- sqrt(mean(errors_ets^2))
    mae_matrix[i, "ETS_Restricted"] <- mean(abs(errors_ets))
  }
  
  # Scale factor for MASE (using ONLY training data - avoids leakage)
  scale_factor <- mean(abs(diff(y[1:first_origin])))
  
  # Average across origins (Approach B from literature)
  results <- data.frame(
    Product = product_name,
    Method = methods,
    ME = round(colMeans(me_matrix), 2),
    RMSE = round(colMeans(rmse_matrix), 2),
    MASE = round(colMeans(mae_matrix) / scale_factor, 2)
  )
  
  return(results)
}

Intermittent Demand Evaluation (Family D)

# ============================================================
# ROLLING ORIGIN EVALUATION FOR INTERMITTENT DEMAND (Family D)
# ============================================================

evaluate_intermittent <- function(data, product_name, origins, h, first_origin) {
  y <- ts(data[[product_name]], frequency = 7)
  
  # Methods include ZERO forecast as critical benchmark
  methods <- c("Naive", "Zero", "Croston", "SBA", "TSB")
  n_methods <- length(methods)
  n_origins <- length(origins)
  
  # Initialize matrices
  rmse_matrix <- matrix(NA, nrow = n_origins, ncol = n_methods,
                        dimnames = list(paste0("o", origins), methods))
  me_matrix <- matrix(NA, nrow = n_origins, ncol = n_methods,
                      dimnames = list(paste0("o", origins), methods))
  mae_matrix <- matrix(NA, nrow = n_origins, ncol = n_methods,
                       dimnames = list(paste0("o", origins), methods))
  
  for (i in seq_along(origins)) {
    origin <- origins[i]
    train <- y[1:origin]
    actual <- as.numeric(y[(origin + 1):(origin + h)])
    
    # ----- NAIVE -----
    fc_naive <- as.numeric(naive(train, h = h)$mean)
    
    # ----- ZERO FORECAST (Critical Benchmark) -----
    # For highly intermittent data, zero forecast may be optimal!
    fc_zero <- rep(0, h)
    
    # ----- CROSTON -----
    # Uses odds-ratio for occurrence probability
    fc_croston <- as.numeric(forecast(adam(train, model = "MNN",
                      occurrence = "odds-ratio", silent = TRUE), h = h)$mean)
    
    # ----- SBA (Syntetos-Boylan Approximation) -----
    # Bias-corrected version of Croston
    fc_sba <- as.numeric(forecast(adam(train, model = "MNN",
                  occurrence = "inverse-odds-ratio", silent = TRUE), h = h)$mean)
    
    # ----- TSB (Teunter-Syntetos-Babai) -----
    # Models demand probability directly
    fc_tsb <- as.numeric(forecast(adam(train, model = "MNN",
                  occurrence = "general", silent = TRUE), h = h)$mean)
    
    # Calculate metrics
    forecasts <- list(fc_naive, fc_zero, fc_croston, fc_sba, fc_tsb)
    for (m in 1:n_methods) {
      errors <- actual - forecasts[[m]]
      me_matrix[i, m] <- mean(errors)
      rmse_matrix[i, m] <- sqrt(mean(errors^2))
      mae_matrix[i, m] <- mean(abs(errors))
    }
  }
  
  # Scale factor for MASE
  scale_factor <- mean(abs(diff(y[1:first_origin])))
  
  results <- data.frame(
    Product = product_name,
    Method = methods,
    ME = round(colMeans(me_matrix), 2),
    RMSE = round(colMeans(rmse_matrix), 2),
    MASE = round(colMeans(mae_matrix) / scale_factor, 2)
  )
  
  return(results)
}

Evaluation Setup

  • Training window: Expanding (starts at 49 days, grows to 63)
  • Forecast horizon: 7 days (matches lead time)
  • Number of origins: 15
  • Metrics: ME (bias), RMSE (accuracy), MASE (relative)

3. Method Selection Framework

# ============================================================
# HIERARCHICAL METHOD SELECTION FRAMEWORK
# ============================================================

select_method <- function(results_df, mean_demand, family, aicc_values = NULL) {
  # GATE 1: Bias Filter
  # Exclude methods where |ME| > 10% of mean demand
  # EXCEPTION: Family D exempt (zeros create stochastic bias)
  
  if (family != "D") {
    bias_threshold <- mean_demand * 0.10
    results_df <- results_df %>%
      filter(abs(ME) <= bias_threshold)
    
    if (nrow(results_df) == 0) {
      warning("All methods failed bias filter!")
      return(NULL)
    }
  }
  
  # GATE 2: Accuracy Selection
  # Select method with lowest RMSE
  min_rmse <- min(results_df$RMSE)
  candidates <- results_df %>%
    filter(RMSE <= min_rmse * 1.05)  # Within 5% of best
  
  # GATE 3: Complexity Tie-Breaker
  # If multiple methods within 5% RMSE, prefer lower AICc
  if (nrow(candidates) > 1 && !is.null(aicc_values)) {
    candidates$AICc <- aicc_values[candidates$Method]
    selected <- candidates %>%
      arrange(AICc) %>%
      slice(1)
    selection_reason <- "AICc tie-breaker"
  } else {
    selected <- candidates %>%
      arrange(RMSE) %>%
      slice(1)
    selection_reason <- "Lowest RMSE"
  }
  
  selected$SelectionReason <- selection_reason
  return(selected)
}

# ============================================================
# RUN EVALUATION FOR ALL PRODUCTS
# ============================================================

# Parameters
h <- 7                    # Forecast horizon (1-week lead time)
first_origin <- 49        # First origin point
origins <- 49:63          # 15 rolling origins

# Store all results
all_results <- list()

# Families A, B, C: Standard methods
for (family in c("A", "B", "C")) {
  for (product in products[[family]]) {
    results <- evaluate_standard(data, product, origins, h, first_origin)
    mean_demand <- mean(data[[product]])
    
    selected <- select_method(results, mean_demand, family)
    all_results[[product]] <- selected
  }
}

# Family D: Intermittent methods
for (product in products$D) {
  results <- evaluate_intermittent(data, product, origins, h, first_origin)
  mean_demand <- mean(data[[product]])
  
  selected <- select_method(results, mean_demand, "D")
  all_results[[product]] <- selected
}

Gate 1: Bias Filter

Exclude methods where |ME| > 10% of mean demand

Rationale: Systematic bias leads to persistent over/understocking regardless of accuracy.

Exception: Family D exempt — zeros create stochastic bias, not systematic.

Gate 2: Accuracy Selection

Among remaining methods, select the one with lowest RMSE

Rationale: RMSE directly links to safety stock calculation (SS = z × RMSE).

Gate 3: Complexity Tie-Breaker

If RMSE within 5%, prefer lower AICc (parsimony principle)

Rationale: Simpler models are more robust and interpretable.

4. Safety Stock Calculation

# ============================================================
# SAFETY STOCK CALCULATION
# ============================================================

# Service level parameters
service_level <- 0.97           # 97% cycle service level
z <- qnorm(service_level)       # z = 1.88

cat("Service level:", service_level, "\n")
cat("z-factor:", round(z, 2), "\n")
# Service level: 0.97
# z-factor: 1.88

# Calculate safety stock for each product
# Formula: SS = ceiling(z × RMSE)
# ceiling() ensures integer units and rounds up for safety

final_results <- do.call(rbind, all_results) %>%
  mutate(
    SafetyStock = ceiling(z * RMSE),
    Family = substr(Product, 1, 1)
  )

# Add mean demand for context
final_results <- final_results %>%
  rowwise() %>%
  mutate(
    MeanDemand = mean(data[[Product]]),
    SS_Pct = round(SafetyStock / MeanDemand * 100, 1)
  ) %>%
  ungroup()

# Summary by family
final_results %>%
  group_by(Family) %>%
  summarise(
    Avg_RMSE = round(mean(RMSE), 1),
    Avg_SS = round(mean(SafetyStock), 0),
    Avg_SS_Pct = round(mean(SS_Pct), 1),
    Methods = paste(unique(Method), collapse = ", ")
  )

5. Key Insights

Simple Methods Win

Moving Average won for 6 of 10 products in Families A and C. Complexity doesn't always improve accuracy. MA is also more interpretable for stakeholders.

Family D Needs Special Treatment

Standard methods fail for intermittent demand. Croston and SBA are essential for products with many zero-demand periods.

Bias Before Accuracy

A method with low RMSE but high bias will systematically over/understock. The bias gate prevents this.

RMSE → Inventory Cost

Using RMSE as the primary selection criterion ensures the chosen method minimizes safety stock while maintaining service level.

High SS Isn't Always Bad

Family D's 327% SS reflects fundamental demand uncertainty, not method failure. Some demand is inherently unpredictable.

MASE for Flagging, Not Selection

High MASE (≥1) flags products needing investigation but shouldn't exclude methods — it can reflect demand difficulty.

6. Limitations & Recommendations

  • Normality assumption: z-factor assumes normally distributed errors; may not hold for Families C and D with skewed demand
  • Economic considerations: For Family D products with SS > 300% of mean, alternative strategies (build-to-order, lower service level) may be more economical
  • Lead time variability: Fixed lead time assumed; if lead time varies, safety stock should be adjusted accordingly
  • Sample size: 70 days is relatively short for reliable intermittent demand estimation
